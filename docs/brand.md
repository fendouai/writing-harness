# Writing Harness Brand Guide

This is a lightweight style guide for README visuals, badges, and future brand assets.

The goal is not heavy branding. The goal is consistency.

## Brand Positioning

Writing Harness should feel:

- editorial, not flashy
- technical, not sterile
- trustworthy, not corporate
- practical, not speculative

The visual system should support one core message:

> This is a serious writing-quality tool, not a cosmetic AI text trick.

## Voice

Use language that is:

- clear
- precise
- lightly opinionated
- calm
- high-signal

Avoid language that feels:

- overhyped
- startup-generic
- overly academic
- cute or gimmicky

Preferred framing:

- authored quality
- writing system
- guides and sensors
- benchmark loop
- repeatable quality
- clearer, sharper, more useful writing

Avoid overusing:

- magic
- effortless
- revolutionary
- seamless
- human-like

## Color Palette

Primary ink:

- `#10233F`

Primary warm accent:

- `#E66A1C`

Primary green accent:

- `#0F9D8A`

Support blue:

- `#1D4F94`

Warm surface:

- `#FAF1E6`
- `#FFF0DE`

Cool surface:

- `#E8F6F1`
- `#E8EFFA`

Neutral text:

- `#43536B`
- `#47607A`

## Typography

For display / headline use:

- `Georgia`
- `"Times New Roman"`

For UI / labels / support text use:

- `"Trebuchet MS"`
- `Verdana`
- sans-serif fallback

This combination should feel editorial plus practical, not generic SaaS.

## Visual Rules

1. Prefer warm editorial backgrounds over flat white.
2. Use rounded cards and soft shadows, not sharp enterprise panels.
3. Keep diagrams readable at GitHub README width.
4. Make every visual answer one product question fast.
5. Use no more than 3-4 main colors in a single asset.

## Asset Types

Recommended asset categories:

- wordmark
- comparison visual
- feature visual
- workflow visual
- social preview

## Badge Rules

Badges should be minimal and useful.

Good badge topics:

- license
- skill name
- benchmark style
- eval status

Avoid:

- vanity badges
- star counts in the hero
- overly bright default badge colors

## README Hero Rules

The top of the README should scan in this order:

1. Name
2. One-line promise
3. Main visual
4. Core framing
5. Badge row
6. Use-case strip
7. What you get

Keep the first screen focused on product value, not methodology depth.
